# sdd-stakeholder — ステークホルダー分析・RACI生成（SDD Phase 0）

## 0. 目的

すべての仕様書作成の"ゼロ番目"として、**誰のための何を作るか**を確定する。
- ステークホルダーマップ（全関係者・影響度・関与度）を生成する
- Power/Interest Grid を Mermaid 図で可視化する
- RACI Matrixで意思決定の責任者を明確化する
- 各ステークホルダーの「期待する価値」「許容できないリスク」「成功基準」を整理する
- ステークホルダーインタビューテンプレートで情報収集を体系化する
- 変更影響評価（Change Impact Assessment）で移行リスクを事前把握する
- 後続の `sdd-context`・`sdd-req100` へのインプットを確立する

**世界標準**: ISO 29148:2018 §5.2 Stakeholder Needs, Volere Iceberg Model, Mendelow Power/Interest Grid

## 1. 入力と出力（ファイル契約）

### 入力
- /sdd-stakeholder $ARGUMENTS
  - $0 = spec-slug（例: google-ad-report）
  - $1 = target-dir（任意。未指定なら `.kiro/specs/<spec-slug>/` を使う）

### 出力（必須）
- `<target-dir>/stakeholder-map.md`

### 参照（既存ファイルがあれば優先）
- 既存の README / docs / 事業計画書 / 提案書 / 議事録

## 2. 重要ルール（絶対）

- ステークホルダーは「社内」「社外」「システム」の3カテゴリで整理する
- 影響度 × 関与度でステークホルダーを4象限にプロットする（Power/Interest Grid）
- Volere Iceberg Model に従い、「表層ニーズ」と「根本ニーズ」を必ず分離する
- RACI Matrixの R（責任者）は必ず1人（複数禁止）
- 未確定のステークホルダーは「TBD」として明記し、スコアを下げる（100点は付けない）
- **利害対立は「対立なし」であっても「確認済み・対立なし」と明記する（スキップ禁止）**
- **変更影響評価（CIA）は必ず実施する**（既存業務・システムへの影響を定量的に記述）

## 3. 手順（アルゴリズム）

### Pre-Phase: 既存ドキュメント確認ゲート

実行前に以下を確認する:

1. README.md, docs/, 事業計画書, 提案書の存在確認（Glob/Read）

**ドキュメントが少ない場合**:
```
⚠️ 警告: プロジェクトの背景資料が不足しています。

推奨:
  - README.md または docs/ に事業背景・目的を記述する
  - または以下の質問にお答えください

このまま続けますか？（情報不足のままステークホルダーを仮定します）
```

### Step A: ターゲットディレクトリ決定
- target-dir = $1 があればそれ、なければ `.kiro/specs/$0/`
- 無ければ作成する

### Step B: コンテキスト収集
既存ファイルを探索して読む（Glob/Grep/Read）:
```
- README.md, docs/, 提案書, 議事録, 事業計画書
- 既存の requirements.md があれば読む
```

情報が不足している場合、以下の質問を最大10個提示してユーザーの回答を待つ:

#### ステークホルダー確認質問（情報不足時のみ）
1. このシステムを最終的に「使う」のは誰ですか？（日常的な利用者）
2. このシステムの「成功・失敗」を判断するのは誰ですか？（意思決定者）
3. このシステムの「予算・リソース」を承認するのは誰ですか？
4. このシステムの影響を受けるが直接使わない人は誰ですか？（被影響者）
5. 法的・規制的な観点で関与する部門・組織はありますか？（コンプライアンス）
6. 外部ベンダー・パートナーで連携が必要な組織はありますか？
7. システムの「失敗」で最も大きなダメージを受けるのは誰ですか？
8. ステークホルダー間に利害対立がありますか？（どこに？）
9. 開発チーム・運用チームの構成は？（内部ステークホルダー）
10. 将来追加されうるステークホルダーはいますか？

ユーザーが「仮置きで進めて」と言った場合は業界標準の仮定で埋め、「前提/仮定」に明記する。

### Step C: `stakeholder-map.md` を生成

以下のテンプレートを完全に埋める:

```markdown
# ステークホルダーマップ — {プロジェクト名}

> 生成日: {YYYY-MM-DD}
> スペック: {spec-slug}
> バージョン: 1.0
> 手法: ISO 29148:2018, Volere Iceberg Model, Mendelow Power/Interest Grid

---

## 1. エグゼクティブサマリー

{プロジェクトの目的を1-3文で。誰が、何のために、何を作るか}

**特定したステークホルダー数**: 社内 {N}名 / 社外 {N}名 / システム {N}件
**主要な利害対立**: {あれば概要 / なければ「確認済み・対立なし」}

---

## 2. ステークホルダー一覧

### 2.1 社内ステークホルダー

| ID | 氏名/役割 | 部門 | 関心事（上位3つ） | 影響度(H/M/L) | 関与度(H/M/L) | 象限 |
|----|-----------|------|-----------------|--------------|--------------|------|
| SH-001 | {役割} | {部門} | {関心事1}, {関心事2}, {関心事3} | H/M/L | H/M/L | 管理/満足/情報/把握 |

### 2.2 社外ステークホルダー

| ID | 氏名/役割 | 組織 | 関心事（上位3つ） | 影響度(H/M/L) | 関与度(H/M/L) | 象限 |
|----|-----------|------|-----------------|--------------|--------------|------|
| SH-101 | {役割} | {組織名} | {関心事1}, {関心事2}, {関心事3} | H/M/L | H/M/L | 管理/満足/情報/把握 |

### 2.3 システムステークホルダー（連携システム・データ提供者）

| ID | システム名 | 役割 | 連携方式 | オーナー | 依存度 |
|----|-----------|------|---------|---------|-------|
| SYS-001 | {システム名} | | REST/gRPC/DB/Event | 社内/社外 | H/M/L |

## 3. Power/Interest Grid（4象限マップ）

### Mermaid 図

```mermaid
quadrantChart
    title Stakeholder Power/Interest Grid
    x-axis Low Interest --> High Interest
    y-axis Low Power --> High Power
    quadrant-1 "【管理】密接に管理"
    quadrant-2 "【満足】満足を保つ"
    quadrant-3 "【把握】定期的に報告"
    quadrant-4 "【情報提供】監視する"
    SH-001: [{x}, {y}]
    SH-002: [{x}, {y}]
    SH-101: [{x}, {y}]
```

### 象限別エンゲージメント戦略

| 象限 | ステークホルダー | エンゲージメント戦略 | 連絡頻度 |
|------|--------------|-----------------|---------|
| 【管理】高影響・高関与 | SH-xxx, SH-xxx | 毎回の意思決定に参加させる | 週次以上 |
| 【満足】高影響・低関与 | SH-xxx | 重大事項のみ報告・承認を得る | 月次 |
| 【把握】低影響・高関与 | SH-xxx | 進捗を定期報告し要求を把握する | 隔週 |
| 【情報提供】低影響・低関与 | SH-xxx | ニュースレター/メールで情報共有 | 月次 |

## 4. Volere Iceberg Model（表層ニーズ vs 根本ニーズ）

| ステークホルダーID | 表層ニーズ（言葉にされたこと） | 根本ニーズ（真の目的） | ビジネス価値 | 満足基準 |
|------------------|---------------------------|---------------------|------------|---------|
| SH-001 | {言葉にされた要求} | {その背後にある真の動機} | {定量的価値} | {何が達成されたら満足か} |

## 5. RACI Matrix

| 決定事項/成果物 | {SH-001役割} | {SH-002役割} | {SH-003役割} | {SH-101役割} | 備考 |
|---------------|-------------|-------------|-------------|-------------|------|
| ステークホルダーマップ承認 | A | R | C | I | |
| 要件定義の承認 | A | R | C | I | |
| 設計書の承認 | A | C | R | I | |
| プロトタイプレビュー | I | R | C | C | |
| リリース判断 | A | C | R | I | |
| 運用引き継ぎ承認 | A | I | R | C | |

凡例: R=実行責任(Responsible) / A=説明責任(Accountable) / C=相談先(Consulted) / I=情報共有(Informed)
**Accountable は各行に必ず1名のみ**

|------------------|----------------------|-----------------|---------------|-------|
| SH-001 | {期待値} | {絶対NG事項} | {数値KPI} | H/M/L |

## 7. 利害対立マップ

| 対立A（SH-xxx） | 対立B（SH-xxx） | 対立内容 | 影響範囲 | 調整方針 | 調整者 |
|----------------|----------------|---------|---------|---------|-------|

## 8. ステークホルダーインタビューテンプレート

各ステークホルダーへのインタビューで使用する共通質問セット。

### 共通質問（全ステークホルダー）
1. このシステムが「成功した」と感じるのはどんな状態ですか？（3つ以内）
2. このシステムが「失敗した」と感じるのはどんな状態ですか？（最悪のシナリオ）
3. 現在の課題・痛みを一番大きいもの3つ教えてください
4. このシステムへの関与で最も時間を取られることは何ですか？
5. 他のステークホルダーとの間で「認識のズレ」を感じることはありますか？

### 役割別追加質問（意思決定者向け）
6. このプロジェクトに割けるリソース（人・予算・時間）の上限は？
7. 6ヶ月後にこのシステムを誰かに説明するとしたら、何と言いますか？
8. リリース判断基準（Go/No-Go）は何ですか？

### 役割別追加質問（実作業者向け）
6. 現在の作業フローで最も非効率だと思う部分はどこですか？
7. このシステムに学習コストをかけられる時間は週何時間ですか？
8. 理想のUX/UIはどんなものですか？参考になるシステムはありますか？

---

## 9. 変更影響評価（Change Impact Assessment）

このシステム導入による既存業務・システムへの影響を評価する。

| 影響を受ける対象 | 現在の状態 | 変更後の状態 | 影響度(H/M/L) | 対処が必要なアクション | 担当SH-ID |
|--------------|----------|-----------|-------------|-----------------|---------|
| {業務プロセス名} | {現状} | {変更後} | H/M/L | {移行作業・トレーニング等} | SH-xxx |
| {既存システム名} | {現状} | {廃止/変更} | H/M/L | {データ移行・連携変更} | SYS-xxx |
| {組織/チーム名} | {現状} | {変更後} | H/M/L | {役割変更・スキルアップ} | SH-xxx |

### 移行リスクサマリー

| リスク | 影響度 | 発生確率 | 緩和策 |
|-------|-------|---------|-------|
| {移行期のデータ不整合} | H/M/L | H/M/L | {ダブルエントリ期間を設ける等} |
| {ユーザー抵抗・習熟コスト} | H/M/L | H/M/L | {トレーニング計画} |

|------------------|---------|---------|-------|---------|---------|
| SH-001 | 週次 | 会議/メール | | 進捗・課題・承認依頼 | YYYY-MM-DD |

## 11. 前提・仮定・未解決事項

### 前提（確定済み）
- {前提1}

### 仮定（未検証）
- {仮定1}（検証方法: {方法}、期限: {日付}）

### 未解決事項（Open Questions）
- [ ] {質問1}（判断期限: {日付}、判断者: {氏名/役割}）

---

## 12. 次のステップ

1. このドキュメントを全ステークホルダー（RACI の A/R 者）にレビューしてもらう
2. ステークホルダーインタビューを実施する（Section 8 のテンプレートを使用）
3. `sdd-context` でビジネス目標を整合させる（PR-FAQ作成）
4. `sdd-req100` で要件定義を開始する（RACI の R 者と協働）
5. 変更影響評価（Section 9）の高影響項目から移行計画を策定する
```

### Step D: 品質チェック（自己検証）

生成後に以下を確認する:
- [ ] 全ステークホルダーに影響度・関与度・象限が設定されているか
- [ ] Power/Interest Grid が Mermaid 図で可視化されているか
- [ ] RACI MatrixのAが各行1名か（複数いたら修正する）
- [ ] 表層ニーズと根本ニーズが分離されているか（Iceberg Model）
- [ ] 利害対立が記録されているか（「確認済み・対立なし」も明記）
- [ ] 変更影響評価（CIA）が実施されているか
- [ ] 未解決事項の判断期限と判断者が記載されているか

## 4. 最終応答（チャットに返す内容）

- 特定したステークホルダー数（社内/社外/システム）
- Power/Interest Grid の象限分布（管理/満足/把握/情報提供）
- 主要な利害対立（あれば）と調整方針
- 変更影響評価の高影響項目リスト
- 未解決事項リスト（Open Questions）
- 生成ファイルパス

## 5. 実行例

```bash
/sdd-stakeholder google-ad-report
```
