# sdd-adr — Architecture Decision Record Generator（MADR形式）

## 0. 目的

**設計決定を記録し、将来の参照・変更を可能にする**。なぜその技術を選んだか、何を捨てたかを明文化する。

- 技術的負債の原因（「なぜこうなったのか不明」）を排除する
- 代替案とトレードオフを後から追跡可能にする
- 新メンバーのオンボーディングコストを削減する
- `sdd-design` の設計根拠をADRで補強する
- `sdd-glossary` の命名規則変更時にADRでその理由を記録する

**世界標準**: MADR（Markdown Any Decision Records）, Michael Nygard ADR, Joel Parker Henderson ADR Template

## 1. 入力と出力（ファイル契約）

### 入力
- /sdd-adr $ARGUMENTS
  - $0 = ADRタイトル（例: "JWTを認証方式として採用"）
  - $1 = spec-slug（例: google-ad-report）
  - $2 = target-dir（任意。未指定なら `.kiro/specs/<spec-slug>/adr/` を使う）

### 入力ファイル（あれば読む）
- `<spec-dir>/requirements.md`（関連する要件 REQ-xxx）
- `<spec-dir>/design.md`（設計との整合確認）
- `<spec-dir>/glossary.md`（用語の一貫性確認）
- `<spec-dir>/adr/`（既存ADR番号確認）

### 出力（必須）
- `<target-dir>/ADR-{NNN}-{slug}.md`（NNN: ゼロ埋め3桁）
- `<target-dir>/README.md`（ADR一覧。存在すれば更新、なければ作成）

## 2. 重要ルール（絶対）

- **MADR形式必須**: 標準化されたセクション構造を厳守
- **代替案は最低3つ**（代替技術A / 代替技術B / 何もしない＝現状維持）
- **各代替案に却下理由を必ず記述する**（「採用した方が良いから」は禁止。数値・根拠を示す）
- **ステータスは Proposed → Accepted の遷移を明記**する
- **関連要件 REQ-xxx を必ず紐付ける**（トレーサビリティ確保）
- ADRは「不変の記録」。後で変更する場合は「Superseded by ADR-XXX」として旧ADRを閉じ、新ADRを作成する

## 3. ADRステータス定義

| Status | 意味 | 遷移条件 |
|--------|------|---------|
| **Proposed** | 提案中・レビュー待ち | 初期作成時 |
| **Accepted** | 承認済み・実装可能 | ステークホルダー承認後 |
| **Deprecated** | 非推奨・新規利用禁止 | より良い方法が確立後 |
| **Superseded** | 後続ADRに置き換え済み | 新ADR作成時 |

## 4. 手順（アルゴリズム）

### Pre-Phase: 入力確認ゲート

実行前に以下を確認する:

1. `<spec-dir>/design.md` の存在確認（Glob/Read）
2. `<spec-dir>/requirements.md` の存在確認（Glob/Read）

**ファイルが存在しない場合**:
```
⚠️ 警告: design.md / requirements.md が見つかりません。

推奨順序:
  /sdd-req100 {spec-slug}         - 要件定義
  /sdd-design {spec-slug}         - アーキテクチャ設計
  /sdd-adr "{title}" {spec-slug}  - 技術決定記録（本スキル）

このまま続けますか？（前提情報なしで記録します）
```

### Step A: ADR番号の決定

1. `<target-dir>/` を Glob でスキャン（`ADR-*.md`）
2. 既存の最大番号 + 1 を新ADR番号とする（初回は 001）
3. タイトルからスラグ生成: 全角→半角、スペース→ハイフン、小文字化
4. ファイル名: `ADR-{NNN}-{kebab-case-slug}.md`

### Step B: コンテキスト収集

情報が不足している場合、以下の質問を提示してユーザーの回答を待つ:

#### ADR記録質問（情報不足時のみ）
1. この決定を迫られた背景・制約（技術的/ビジネス的/時間的）は何ですか？
2. 今回選択した技術/アプローチの名称は？
3. 最終的にこれを選んだ理由を3つ以上挙げてください
4. 検討した代替案は何ですか？（最低2つ）
5. 代替案を却下した理由は？（数値・根拠付き）
6. この決定によって生まれる負の影響・リスクは？
7. 関連する要件番号（REQ-xxx）はありますか？
8. この決定のレビュー・見直し予定日はありますか？
9. 承認者（ステークホルダー）は誰ですか？
10. この決定が将来覆る可能性がある条件は？

ユーザーが「仮置きで進めて」と言った場合は業界標準の仮定で埋め、「前提/仮定」に明記する。

### Step C: `ADR-{NNN}-{slug}.md` を生成

以下のテンプレートを完全に埋める:

```markdown
# ADR-{NNN}: {タイトル}

> 作成日: {YYYY-MM-DD}
> ステータス: Proposed
> スペック: {spec-slug}
> 著者: {著者 or "未設定"}
> 承認者: {承認者 or "未設定"}
> レビュー予定: {日付 or "未定"}

---

## ステータス遷移

```
Proposed → Accepted (YYYY-MM-DD)
```

(もし Superseded の場合)
```
Superseded by [ADR-{NNN}: {タイトル}](./{filename})
```

### Step E: 品質チェック（自己検証）

生成後に以下を確認する:
- [ ] ADR番号が既存と重複していないか
- [ ] 代替案が最低3つ（うち1つは「何もしない」）記載されているか
- [ ] 各代替案に具体的な却下理由が記載されているか
- [ ] 関連要件 REQ-xxx が紐付けられているか
- [ ] 比較表の全セルが埋まっているか
- [ ] 成功基準に定量的な指標が含まれているか
- [ ] adr/README.md が更新されているか

## 5. 最終応答（チャットに返す内容）

- 生成したADRファイルパス（ADR番号付き）
- ステータス（Proposed / Accepted）
- 検討した代替案数
- 関連要件数
- 次のアクション（誰が承認するか、いつ実装するか）

## 6. 実行例

```bash
/sdd-adr "JWTを認証方式として採用" google-ad-report
```

出力:
```
.kiro/specs/google-ad-report/adr/
├── ADR-001-jwt-authentication.md
└── README.md
```
